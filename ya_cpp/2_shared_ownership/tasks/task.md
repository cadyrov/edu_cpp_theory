# Задания по совместному владению

## 1. std::shared_ptr и std::weak_ptr

1. Создайте объект через `std::make_shared`.
2. Скопируйте `shared_ptr` и проверьте `use_count`.
3. Передайте `shared_ptr` в функцию по значению.
4. Передайте `shared_ptr` в функцию по `const&`.
5. Создайте `weak_ptr` из `shared_ptr`.
6. Получите объект из `weak_ptr` через `lock`.
7. Проверьте поведение `lock` после уничтожения последнего `shared_ptr`.
8. Напишите класс, который логирует деструктор, и проверьте момент удаления.
9. Покажите, что `weak_ptr` не продлевает жизнь объекта.
10. Напишите пример, где `shared_ptr` действительно нужен из-за нескольких владельцев.

## 2. Опасности и ограничения std::shared_ptr

1. Напишите пример лишнего копирования `shared_ptr`.
2. Исправьте его передачей по `const&`.
3. Напишите пример, где для чтения лучше подходит `const T&`.
4. Покажите ошибочный пример двух `shared_ptr` из одного сырого указателя.
5. Покажите, почему `shared_ptr<T>(this)` опасен.
6. Создайте цикл владения из двух объектов с `shared_ptr`.
7. Исправьте цикл через `weak_ptr`.
8. Напишите пример, где `use_count()` дает хрупкую бизнес-логику.
9. Покажите, что `shared_ptr` не делает сам объект потокобезопасным.
10. Перепишите пример с `shared_ptr` на `unique_ptr`, если владелец один.

## 3. std::shared_ptr в многопоточной среде

1. Скопируйте один `shared_ptr` в несколько потоков.
2. Проверьте, что объект живет до завершения всех копий.
3. Напишите пример гонки данных внутри объекта под `shared_ptr`.
4. Исправьте гонку через `std::mutex`.
5. Передайте `shared_ptr<const T>` читателям.
6. Используйте `std::atomic<std::shared_ptr<T>>` для публикации объекта.
7. Реализуйте чтение через локальную копию `shared_ptr`.
8. Реализуйте замену текущего объекта новым.
9. Проверьте, что старые читатели продолжают видеть старую версию.
10. Объясните разницу между потокобезопасностью control block и самого объекта.

## 4. Case study: динамически обновляемая конфигурация

1. Опишите структуру `Config` с несколькими полями.
2. Храните текущую конфигурацию в `std::shared_ptr<const Config>`.
3. Напишите функцию чтения текущей конфигурации.
4. Напишите функцию обновления, создающую новый `Config`.
5. Запретите изменение уже опубликованной конфигурации.
6. Добавьте атомарную публикацию новой версии.
7. Напишите несколько читателей, которые берут локальную копию `shared_ptr`.
8. Проверьте, что старый читатель не ломается при обновлении.
9. Уберите возврат сырой ссылки на текущую конфигурацию.
10. Добавьте поле `version` и выведите, какую версию видит каждый читатель.

## 5. Семантика перемещения в стандартной библиотеке

1. Переместите `std::unique_ptr` и проверьте исходное состояние.
2. Переместите `std::shared_ptr` и проверьте исходное состояние.
3. Переместите `std::string` и присвойте ей новое значение после move.
4. Переместите `std::vector<int>` и не читайте старое содержимое.
5. Переместите `std::vector<std::unique_ptr<int>>`.
6. Проверьте, что `const std::unique_ptr` нельзя нормально переместить.
7. Сравните move для `std::vector` и `std::array`.
8. Напишите функцию, принимающую контейнер по значению и вызываемую через `std::move`.
9. Напишите пример, где `std::move` не очищает объект.
10. Сформулируйте правило безопасной работы с moved-from объектом.

## 6. Взаимодействие с API-библиотекой

1. Напишите C-style функцию, которая принимает `T*` без владения.
2. Передайте туда `unique_ptr.get()`.
3. Напишите C-style функцию, которая принимает владение сырым указателем.
4. Передайте туда ресурс через `release`.
5. Напишите custom deleter для ресурса из C API.
6. Оберните `FILE*` в `std::unique_ptr` с `fclose`.
7. Напишите callback, который получает сырой указатель без продления жизни.
8. Перепишите callback с захватом `std::shared_ptr`.
9. Перепишите callback с захватом `std::weak_ptr` и проверкой `lock`.
10. Напишите адаптер для API, которое создает ресурс через `T**`.
